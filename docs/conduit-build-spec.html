Epic 0 — Project Foundation & Architecture

0 / 6

Scaffold the app shell, module structure, routing, and the API/auth plumbing that every other feature depends on.

  • As a developer I want a clean NgModule architecture with CoreModule, SharedModule, and lazy-loaded feature modules so that the codebase mirrors a classic enterprise Angular structure and features stay decoupled.
  • Acceptance criteria
  • Given a fresh Angular CLI workspace, when the app is generated, then the tree contains AppModule, CoreModule (singletons: services, interceptors, guards, header/footer), SharedModule (declarations of reusable components/pipes/directives, re-exporting CommonModule and ReactiveFormsModule), and separate feature modules (auth, home, article, editor, profile, settings).
  • Given CoreModule, when it is imported a second time, then its constructor throws via the @Optional() @SkipSelf() guard so it can only be imported once by AppModule.
  • Given the routing config, when the user navigates to /login, /register, /editor, /settings, /article/:slug, or /profile/:username, then the corresponding feature module is lazy-loaded via loadChildren (verified by a separate JS chunk per feature in the build output).
  • Given SharedModule, when a feature module needs a shared component (e.g. ArticlePreview, ArticleMeta, FavoriteButton), then it imports SharedModule rather than re-declaring, and SharedModule declares nothing that provides a service.
  • Given the app boots, when the root route / is hit, then the home feature loads and renders inside the persistent shell without a full reload.
  • Angular concepts exercised — NgModule architecture (Core/Shared/Feature split), RouterModule.forRoot(routes) in the root routing module vs RouterModule.forChild(routes) in each feature routing module, lazy loading via loadChildren (one chunk per feature), the @Optional() @SkipSelf() CoreModule import guard, and the SharedModule re-export pattern (exports of CommonModule/ReactiveFormsModule with no providers).
  • Hints — Keep providers out of SharedModule to avoid duplicate instances in lazy chunks; put all singletons in CoreModule (or providedIn: 'root') and re-export CommonModule/ReactiveFormsModule/RouterModule from SharedModule.
  • As a developer I want a single ApiService that wraps HttpClient with the API base URL supplied by environment/InjectionToken so that feature services call get/post/put/delete with relative paths and the base URL is never hardcoded.
  • Acceptance criteria
  • Given environment.ts / environment.prod.ts, when the app builds, then the base URL https://api.realworld.show/api is defined there and provided through an API_BASE_URL InjectionToken in CoreModule.
  • Given ApiService, when a feature service calls api.get<T>('/tags'), then the request goes to https://api.realworld.show/api/tags and returns a typed Observable<T>.
  • Given api.get('/articles', params), when params { tag, author, favorited, limit, offset } are passed, then they are serialized as query string via HttpParams, omitting undefined keys.
  • Given all four verbs, when get/post/put/delete are called, then each returns a typed observable and body/params are typed generically (no any leaking to callers).
  • Given HttpClientModule, when the app boots, then it is imported exactly once (in CoreModule/AppModule), not in feature modules.
  • Angular concepts exercised — HttpClient constructor injection, defining an InjectionToken<string> for API_BASE_URL provided with useValue from environment, generic wrapper methods returning Observable<T>, building query strings immutably with HttpParams (each .set() returns a new instance), and root-scoped DI via @Injectable({ providedIn: 'root' }).
  • Hints — Build HttpParams by reducing over the params object and skipping null/undefined; keep ApiService transport-only — no domain logic.
  • As a developer I want an HTTP_INTERCEPTORS interceptor that reads the JWT from localStorage and sets Authorization: Token <jwt> so that authenticated endpoints work without every service manually adding the header.
  • Acceptance criteria
  • Given a stored JWT, when any request leaves HttpClient, then the interceptor clones the request and adds header Authorization: Token <jwt>.
  • Given no token in localStorage (logged-out user), when a request is sent (e.g. GET /articles, GET /profiles/:username), then no Authorization header is attached so auth-optional endpoints still succeed.
  • Given a JwtService (or AuthService) that owns token read/write/clear, when the interceptor needs the token, then it reads it from that service and never touches localStorage directly.
  • Given the interceptor, when it is registered, then it is provided via { provide: HTTP_INTERCEPTORS, useClass: JwtInterceptor, multi: true } in CoreModule.
  • Given login/registration responses (POST /users/login, POST /users), when they return { user: { token } }, then the token is persisted and immediately used by subsequent requests without a reload.
  • Angular concepts exercised — Implementing HttpInterceptor.intercept(req, next) and returning next.handle(...), treating HttpRequest as immutable via req.clone({ setHeaders }), registering with the HTTP_INTERCEPTORS multi-provider (multi: true), and constructor DI of the token service so the token is read per-request inside intercept.
  • Hints — Interceptors are singletons resolved once — inject the token service in the constructor and read the token per-request inside intercept, not at construction time.
  • As a developer I want a global error interceptor that centralizes HTTP failure handling so that 401s log the user out and other errors surface consistently instead of being handled ad hoc per call.
  • Acceptance criteria
  • Given any request, when the response is a 401 Unauthorized, then the interceptor purges the stored token and redirects to /login via the Router.
  • Given a 422 Unprocessable Entity from the API (RealWorld's { errors: { field: [msgs] } } shape), when it is returned, then the error is re-thrown in a normalized shape that forms can bind to (field to messages).
  • Given a 5xx or network error, when it occurs, then the interceptor re-throws a normalized error usable by the UI (e.g. a toast/banner message).
  • Given the error stream, when an error passes through, then it is handled with catchError and re-emitted via throwError(() => ...) so downstream subscribers still receive it.
  • Given registration order, when interceptors run, then the error interceptor and JWT interceptor coexist and both apply to the same request (interceptors run in registration order on the outgoing request and in reverse order on the response).
  • Angular concepts exercised — A second HttpInterceptor chained via a multi: true HTTP_INTERCEPTORS provider, piping next.handle(req) through RxJS catchError and re-emitting with throwError(() => normalizedError), narrowing failures via instanceof HttpErrorResponse and branching on err.status, and performing a Router.navigate(['/login']) side-effect from inside the interceptor.
  • Hints — Use catchError in the returned pipe and branch on err.status; normalize the RealWorld errors map into a flat list before re-throwing so form components stay simple.
  • As a user I want a persistent header and footer whose nav links change based on whether I am logged in so that I always see the actions relevant to my auth state.
  • Acceptance criteria
  • Given a logged-out user, when any page renders, then the header shows Home, Sign in (/login), and Sign up (/register) links.
  • Given a logged-in user, when any page renders, then the header shows Home, New Article (/editor), Settings (/settings), and the user's profile link (/profile/:username, with avatar/username from GET /user).
  • Given app startup, when a token exists in localStorage, then GET /user is called once to populate the current-user state before nav renders the authed links; and if that call fails/401s, the app treats the user as logged out.
  • Given the current-user state, when the user logs in or out, then the header updates reactively without a page reload (driven by an observable, not a manual DOM refresh).
  • Given router navigation, when the active route matches a nav link, then that link receives the active class via routerLinkActive.
  • Given the layout, when any feature page loads, then it renders inside a <router-outlet> between the shared header and footer.
  • Angular concepts exercised — AuthService holding current-user state in a BehaviorSubject and exposing currentUser$/isAuthenticated$ observables, consuming them in the header template with the async pipe (auto-subscribe/unsubscribe) plus *ngIf ... as for the two nav states, an APP_INITIALIZER factory whose returned Promise/Observable Angular awaits before bootstrap to hydrate the current user, and router-outlet with routerLink + routerLinkActive for active-link styling.
  • Hints — Expose currentUser$ and isAuthenticated$ from AuthService; consume with the async pipe so subscriptions auto-clean. Register an APP_INITIALIZER factory that returns the GET /user call as a Promise or Observable — Angular waits for it to complete before first render (and swallow its error so a bad/expired token doesn't block bootstrap).
  • As a user I want a friendly 404 page for unknown URLs so that bad links don't leave me on a blank screen.
  • Acceptance criteria
  • Given any URL that matches no defined route, when it is navigated to, then a NotFoundComponent renders inside the persistent shell with a message and a link back to Home (/).
  • Given the root routing table, when routes are declared, then the wildcard { path: '**', component: NotFoundComponent } is the last entry so it never shadows real routes.
  • Given a lazy feature route with a bad child path (e.g. /profile with no username), when it fails to match, then the wildcard catches it and shows the 404 rather than erroring.
  • Given the 404 page, when it renders, then the header/footer remain visible and the user can navigate away without a reload.
  • Angular concepts exercised — The wildcard route { path: '**', component: NotFoundComponent }, route-matching order in RouterModule.forRoot (Angular matches top-down, first-match-wins, so ** must be declared last), routerLink for recovery navigation back to /, and rendering the component through the shared shell's router-outlet.
  • Hints — Keep path: '**' last in the top-level Routes array; a plain component (no guard/resolver) is enough.

Epic 1 — Authentication & Session

0 / 6

Let users register, sign in, persist their session across reloads, and sign out, with the header reflecting auth state reactively.

  • As a new visitor I want to create an account from /register so that I can post articles and follow authors.
  • Acceptance criteria
  • Given I am on /register, when the page renders, then I see a reactive form with username, email, and password controls and a disabled "Sign up" button.
  • Given the form is invalid (empty required fields, malformed email, password shorter than the min length), when I blur or edit a control, then inline validation messages appear and the submit button stays disabled.
  • Given a valid form, when I submit, then a POST /users is sent with { user: { username, email, password } }, the button shows a submitting/disabled state, and a duplicate submit is prevented.
  • Given the request succeeds, when { user: { token, ... } } is returned, then the token is written to localStorage, session state is populated, and I am navigated to /.
  • Given the server returns 422 with a { errors: { field: string[] } } body (e.g. email: ["has already been taken"]), when the response arrives, then each field error is rendered in an "error messages" list and the form remains editable.
  • Given a network/500 error, when it occurs, then a generic error is shown and the button is re-enabled.
  • Given I already have a valid session, when I navigate to /register, then I am redirected to /.
  • Angular concepts exercised — Reactive forms built with FormBuilder.group, Validators.required/Validators.email/Validators.minLength, per-control error rendering gated on control.touched/control.dirty, HttpClient.post<T> with a typed { user: User } response, RxJS catchError to surface server errors and finalize to reset the submitting flag, constructor DI of AuthService, Router.navigate(['/']) on success.
  • Hints — Map the RealWorld { errors: { field: string[] } } shape into a flat display list; use finalize to reset the submitting flag on both success and error.
  • As a returning user I want to sign in from /login so that I can access my account and authenticated features.
  • Acceptance criteria
  • Given I am on /login, when the page renders, then I see a reactive form with email and password controls and a "Sign in" button disabled while the form is invalid.
  • Given a valid form, when I submit, then a POST /users/login is sent with { user: { email, password } } and the button enters a disabled submitting state.
  • Given the request succeeds, when the response returns { user: { token, ... } }, then the token is persisted to localStorage, session state updates from the same response, and I am navigated to /.
  • Given credentials are wrong, when the server returns 422 with { errors: { "email or password": ["is invalid"] } }, then the error list is displayed and inputs stay filled and editable.
  • Given I mistype and correct a field, when validity changes, then the submit button enables/disables reactively without a page reload.
  • Given I already have a valid session, when I navigate to /login, then I am redirected to /.
  • Angular concepts exercised — Reactive forms, HttpClient.post<{ user: User }>, tap to persist the token and push the returned user into the session BehaviorSubject (the login response already includes the user, so no second request is chained), shared AuthService state, Router redirect on success, catchError mapping the 422 body to component error state.
  • Hints — Reuse a single AuthService.authenticate(payload, endpoint) method for both login and register to avoid duplicated token-persistence logic; the response shape is identical, so both paths just tap the returned user into the session subject.
  • As a signed-in user I want my session restored after a full page reload so that I stay logged in without re-entering credentials.
  • Acceptance criteria
  • Given a token exists in localStorage, when the app bootstraps, then a GET /user is issued with the Authorization: Token <jwt> header before the header/nav settles into its authenticated state.
  • Given GET /user succeeds, when { user } returns, then the reactive session subject emits the user and the header shows authenticated links.
  • Given no token is present, when the app bootstraps, then no GET /user is made and the header shows the anonymous state (Sign in / Sign up).
  • Given the stored token is expired/invalid and GET /user returns 401, when the response arrives, then the token is cleared from localStorage and the session falls back to anonymous.
  • Given the bootstrap GET /user is still in flight, when a guarded route is requested, then the guard waits for the resolved auth state rather than eagerly redirecting.
  • Angular concepts exercised — APP_INITIALIZER provider whose factory returns a function producing a Promise (or an observable converted via firstValueFrom/.toPromise()) that must resolve/complete for bootstrap to proceed; an HttpInterceptor attaching the Authorization: Token <jwt> header; a BehaviorSubject<AuthState> holding the session; catchError(() => of(null)) on 401 to swallow the error so the initializer still resolves; the initializer resolving on both success and error so the app always boots.
  • Hints — Model auth as a tri-state (unknown → authenticated | anonymous) so guards can distinguish "still loading" from "definitely logged out"; resolve the initializer even on error so the app still boots.
  • As a signed-in user I want to log out from the header so that my session is cleared on this device.
  • Acceptance criteria
  • Given I am authenticated, when I click "Logout" in the header, then the token is removed from localStorage and the session subject emits the anonymous state.
  • Given I log out, when the state updates, then the header immediately swaps to anonymous links with no reload, and I am navigated to /.
  • Given I log out while on a guarded route (/settings or /editor), when the session clears, then the guard redirects me to /login.
  • Given I am logged out, when I reload the page, then no GET /user is issued and I remain anonymous.
  • Angular concepts exercised — AuthService.logout() clearing localStorage and calling .next(null) on the session BehaviorSubject; a (click) handler in the header component invoking it; the header rendering from currentUser$ via the async pipe (no manual subscription); Router.navigate(['/']); no server call (RealWorld JWTs are stateless, so logout is a client-side token discard).
  • As a product owner I want authenticated-only routes protected so that anonymous users cannot reach the editor or settings.
  • Acceptance criteria
  • Given I am anonymous, when I navigate to /settings, /editor, or /editor/:slug, then a CanActivate guard blocks activation and redirects me to /login.
  • Given I am authenticated, when I navigate to the same routes, then the guard returns true and the route activates.
  • Given the app is still resolving auth (unknown state on first load with a stored token), when I hit a guarded route directly by URL, then the guard defers its decision until auth resolves, then activates or redirects accordingly.
  • Given the guard redirects to /login, then no protected component is instantiated or rendered before the redirect.
  • Angular concepts exercised — A CanActivateFn (functional guard) using inject(AuthService) and inject(Router), returning Observable<boolean | UrlTree>; filter(state => state !== 'unknown') to wait out bootstrap, take(1) to complete the guard's observable, then map to true or router.createUrlTree(['/login']); route config wiring canActivate: [authGuard].
  • Hints — Return a UrlTree (router.createUrlTree(['/login'])) instead of imperatively navigating; combine filter(state => state !== 'unknown') with take(1) so the guard waits out the bootstrap.
  • As a user I want the header nav to update the instant I log in or out so that the UI always matches my session without a refresh.
  • Acceptance criteria
  • Given I am anonymous, when the header renders, then it shows "Home", "Sign in", and "Sign up".
  • Given I authenticate via A1 or A2, when the session subject emits the user, then the header immediately swaps to "Home", "New Article" (/editor), "Settings" (/settings), and my username/profile link (/profile/:username) — no reload.
  • Given I log out via A4, when the state emits anonymous, then the header reverts to the anonymous links immediately.
  • Given the active route matches a nav item, when it renders, then that item receives an active class.
  • Given the session is still unknown during bootstrap, when the header renders, then it does not flicker between anonymous and authenticated (it renders a stable state until auth resolves).
  • Angular concepts exercised — async pipe subscribing to AuthService.currentUser$ in the template (subscription and teardown handled by the pipe); *ngIf/*ngIf...else (or ngSwitch over the tri-state) selecting the anonymous vs authenticated nav; routerLinkActive="active" for active-link styling; ChangeDetectionStrategy.OnPush, correct because the view is driven purely by the piped observable rather than mutable component fields.
  • Hints — Drive the whole header off a single currentUser$ (or a vm$ combining user + loading flag) so change detection stays OnPush-friendly and there are no manual subscribe/unsubscribe calls.

Epic 2 — Home, Feeds & Tags

0 / 6

Show global and personal article feeds, tag filtering, and pagination on the home page.

  • As a visitor I want to see a paginated list of the most recent articles on the home page so that I can discover content without logging in.
  • Acceptance criteria
  • Given I navigate to / unauthenticated, when the page loads, then the "Global Feed" tab is active and a GET /articles?limit=20&offset=0 request is fired.
  • Given the request is in flight, when no response has returned, then a "Loading articles..." indicator is shown and no article previews are rendered.
  • Given the response returns articles, when it resolves, then each preview shows author image/name, formatted date, title, description, tag list, and a favorite button with the article's favoritesCount.
  • Given the response returns { articles: [], articlesCount: 0 }, when it resolves, then an empty state reads "No articles are here... yet."
  • Given the GET /articles request fails (non-2xx), when it errors, then an inline error message is shown and the loading indicator is cleared.
  • Given the feed is displayed, when I compare it to the API payload, then articles appear in the order returned (no client re-sorting).
  • Angular concepts exercised — HttpClient.get<ArticleListResponse>() typed to the { articles, articlesCount } payload; service method returning Observable<ArticleListResponse>; async pipe subscribing/unsubscribing in the template; ChangeDetectionStrategy.OnPush with trackBy keyed on article.slug; catchError returning a fallback/error emission so the stream stays alive.
  • Hints — Model the feed state as a single { status, articles, count } stream so template branching (loading/error/empty/loaded) is driven by one async subscription rather than multiple flags.
  • As a logged-in user I want a "Your Feed" tab showing articles from authors I follow so that I can keep up with people I care about.
  • Acceptance criteria
  • Given I am unauthenticated, when the home page renders, then the "Your Feed" tab is not shown (only "Global Feed" and any active tag tab appear).
  • Given I am authenticated, when the home page renders, then "Your Feed" is shown as the default active tab and a GET /articles/feed?limit=20&offset=0 request is fired with the Authorization: Token <jwt> header.
  • Given I am on "Your Feed", when I click "Global Feed", then the active tab switches, GET /articles is fired, and offset resets to 0.
  • Given I follow no one, when the feed resolves empty, then the empty state reads "No articles are here... yet."
  • Given a feed request is in flight, when I switch tabs before it resolves, then the stale request is cancelled and only the new tab's result renders.
  • Given the auth header is required, when the feed request is built, then the JWT is attached by the interceptor (not manually in the component).
  • Angular concepts exercised — BehaviorSubject<FeedTab> holding active-tab state (seeded so the feed loads on init); switchMap swapping the source observable so a new tab tears down the in-flight HttpClient request (HttpClient aborts the XHR on unsubscribe); an HttpInterceptor that clones the request to add Authorization: Token <jwt>; *ngIf on authService.currentUser$ (via async) gating the "Your Feed" tab.
  • Hints — Merge tab state and query state into one source stream; use switchMap so tab changes tear down the previous HTTP subscription. Read auth state from an authService.currentUser$ rather than reading localStorage in the component.
  • As a reader I want a sidebar of popular tags so that I can quickly jump to topics that interest me.
  • Acceptance criteria
  • Given the home page loads, when the sidebar initializes, then a GET /tags request is fired once.
  • Given the tags request is in flight, when no response has returned, then the sidebar shows a "Loading tags..." placeholder.
  • Given the response returns { tags: [...] }, when it resolves, then each tag renders as a clickable pill in the "Popular Tags" panel.
  • Given the response returns an empty tag array, when it resolves, then the sidebar shows "No tags are here... yet."
  • Given the GET /tags request fails, when it errors, then the sidebar degrades gracefully (shows no pills, no thrown error blocking the feed).
  • Angular concepts exercised — A dedicated TagsService calling HttpClient.get<{ tags: string[] }>('/tags') and mapping to string[]; async pipe binding the tags$ list; shareReplay({ bufferSize: 1, refCount: true }) so multiple subscribers share one completed request without refetching; OnPush sidebar component; catchError(() => of([])) to degrade to an empty list.
  • Hints — Expose tags$ from the service and bind directly with async; wrap the source in shareReplay({ bufferSize: 1, refCount: true }) so the tag list is shared without a second network call and the upstream subscription is released once no one is listening.
  • As a reader I want clicking a tag to filter the article list to that tag so that I can browse a single topic.
  • Acceptance criteria
  • Given I click a tag pill, when navigation occurs, then the URL updates to /?tag=<name> and a GET /articles?tag=<name>&limit=20&offset=0 request is fired.
  • Given a tag query param is present, when the tabs render, then a third tab labeled #<tag> appears and is active, alongside "Global Feed".
  • Given a tag tab is active, when I click "Global Feed", then the tag param is removed from the URL and the unfiltered GET /articles is fired.
  • Given I land directly on /?tag=<name> (deep link/refresh), when the page loads, then the tag tab is active and the filtered request fires without any extra click.
  • Given switching between "Global Feed", "Your Feed", and a tag, when a request is already in flight, then the previous request is cancelled and offset resets to 0.
  • Angular concepts exercised — Reading ActivatedRoute.queryParamMap and switchMap-ing into the HttpClient call; routerLink="/" with [queryParams]="{ tag: name }" for tag navigation; deriving tab/filter state from the URL as the single source of truth; distinctUntilChanged with an explicit comparator on the extracted tag/offset values (a raw queryParamMap emits a fresh ParamMap reference each time, so default reference equality would never dedupe).
  • Hints — Drive the feed from route.queryParamMap (tag, offset) rather than component state, so deep-linking and back/forward "just work"; map the param map into a plain request descriptor first, dedupe with distinctUntilChanged over that descriptor, then switchMap into the HTTP call.
  • As a reader I want paginated feed results so that I can page through more than 20 articles without an overwhelming single list.
  • Acceptance criteria
  • Given a feed response with articlesCount > limit, when it resolves, then a pager renders ceil(articlesCount / 20) numbered page buttons.
  • Given I click page N, when navigation occurs, then the URL updates to include offset=(N-1)*20 and the matching feed request (GET /articles or GET /articles/feed) fires with that limit/offset.
  • Given a page is active, when the pager renders, then the current page button is visually marked active and clicking it does not refire the request.
  • Given I switch tabs or select a tag, when the new feed loads, then pagination resets to page 1 (offset=0).
  • Given articlesCount <= limit, when the feed resolves, then no pager is rendered.
  • Given a page request is in flight, when I click a different page, then the stale request is cancelled and only the latest page renders.
  • Angular concepts exercised — routerLink with [queryParams]="{ offset }" (and queryParamsHandling: 'merge' so tag is preserved) driving pagination through the URL; deriving offset from queryParamMap into the same switchMap pipeline as tag/tab so filtering and paging compose; computing the page count with Math.ceil(articlesCount / limit) in an OnPush component; trackBy keyed on the page number.
  • Hints — Keep offset in the URL alongside tag so pagination composes with filtering through one queryParamMap stream; compute page numbers from articlesCount in the component, not the template.
  • As a user I want clear loading, empty, and error feedback across every feed variant so that I always know what the app is doing.
  • Acceptance criteria
  • Given any feed request (global, feed, tag, paged) is in flight, when it has not resolved, then a single loading indicator shows and stale results are hidden to prevent flicker.
  • Given a resolved response with zero articles, when it renders, then the "No articles are here... yet." empty state shows and no pager renders.
  • Given a feed request errors, when it fails, then an inline error message shows, the loading indicator clears, and the tabs/tag sidebar remain interactive for retry.
  • Given I rapidly switch tabs/tags, when only the latest request should win, then intermediate results never flash on screen (stale responses discarded).
  • Given the loading, error, empty, and loaded views, when the state stream emits, then exactly one of them is visible at any time.
  • Angular concepts exercised — A discriminated-union view-state ({ kind: 'loading' } | { kind: 'loaded'; articles } | { kind: 'empty' } | { kind: 'error'; message }) modeled as one Observable; startWith({ kind: 'loading' }) placed inside the switchMap so each new request self-resets the view; catchError mapping an HttpErrorResponse into of({ kind: 'error', ... }) so the stream keeps emitting; a single async pipe with *ngSwitch over state.kind; OnPush.
  • Hints — Represent state as { kind: 'loading' | 'loaded' | 'empty' | 'error' } and switch on kind in the template; emit loading via startWith inside the switchMap so each new request self-resets the view.

Epic 3 — Article CRUD & Reading

0 / 5

Let readers view a full article with rendered markdown and let authors create, edit, and delete their own articles via slug-based routing.

  • As a reader I want to open an article by its slug so that I can read the full body, see who wrote it, and browse its tags.
  • Acceptance criteria
  • Given I navigate to /article/:slug, when the component initializes, then it reads slug from the route params and calls GET /articles/:slug (auth optional) to load the article.
  • Given the request is in flight, when the body has not resolved, then a loading state is shown and the article content is not rendered.
  • Given the article resolves, when it renders, then the body markdown is converted to HTML, that HTML is sanitized, and the author's username, avatar (author.image), and formatted createdAt date are shown in the article meta.
  • Given the article has a tagList, when it renders, then each tag is displayed as a non-interactive pill/tag chip.
  • Given the slug does not exist, when GET /articles/:slug returns 404, then an error/not-found state is shown instead of a broken layout.
